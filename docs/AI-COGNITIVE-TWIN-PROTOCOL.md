## 🛡️ 2. The Environmental Boundary (Critical)
The AI is hosted within a **Windows-based terminal environment (PowerShell)**. The project target is **Linux (WSL2/Production)**. To prevent "Execution Hallucinations" and shell mismatches, the following laws are enforced:

### ⚖️ Law of Execution Sovereignty
1.  **Non-Operator Status**: The AI MUST NOT attempt to execute Linux-specific shell scripts, Ansible playbooks, or Podman commands directly.
2.  **The Direct Command Fallback**: Any command requiring a Linux kernel feature MUST be provided as a snippet for the **Human Architect** to execute.
3.  **The 'wsl' Bridge Constraint**: Heavy execution (e.g., Ansible, shell loops, or network tests) MUST NOT be automated via the `wsl` command bridge to prevent session timeouts or empty file errors.
4.  **Proactive Request Mandate**: If Linux-native verification is required, the AI MUST proactively ask the Architect to execute it in the native WSL2 terminal and share the output.

## 📂 4. Feature-First Git Flow
To maintain the absolute integrity of the `main` branch:
1.  **Isolated Development**: All changes MUST be performed in a dedicated feature branch (e.g., `feat/phase-x`).
2.  **Staging & Push**: The AI will stage, commit, and push changes to the feature branch in the Windows host.
3.  **WSL2 Pull**: The Architect will `git pull origin <branch_name>` within WSL2 to verify logic on native Linux.
4.  **The Merge Trigger**: Merging to `main` ONLY occurs after successful verification AND upon explicit request from the Architect.
